For Question 1, I used a transaction ledger instead of a single balance column because I wanted every credit change to be recorded clearly. If I only stored a balance number, I would not know how that number was reached. With a ledger, every credit added or deducted is saved as a separate record with a reason. This makes it easy to track, debug, and fix issues if something goes wrong. It also helps in refunds and audits. The downside is that calculating the balance requires adding up transactions, but I accepted that for better safety and clarity.

For Question 2, I solved the simultaneous deduction problem using database transactions and row-level locking. When a credit deduction starts, the system locks that organization’s credit record so no other request can change it at the same time. If two requests try to deduct 25 credits when only 25 exist, the first one succeeds and updates the balance. The second one waits, then sees there are no credits left and fails. This prevents double spending. I chose this method because it is simple, reliable, and handled directly by the database.

For Question 3, credit deduction and job creation happen inside the same database transaction. That means either both happen or neither happens. If the worker crashes after the credits are deducted, the job is still saved in the database and marked as pending. The system can retry the job safely. I do not immediately refund credits because the job is still valid and can be retried. Refunds only happen if the job fully fails after retries. This keeps credits and job state consistent.

For Question 4, idempotency is handled using an Idempotency-Key stored in the database with a unique constraint. When a request comes in, the system checks if the same key was already used. If yes, it returns the previous result instead of processing again. The unique constraint ensures that even if two identical requests come at the same time, only one is saved. This prevents duplicate charges or duplicate jobs. The check happens before running the main business logic.

For Question 5, at 10 times more load, the database would likely become the main bottleneck, especially because of locking during credit deduction. More users means more waiting on the same rows. To fix this, I would improve indexing, reduce unnecessary queries, and possibly split data by organization to reduce lock conflicts. I would also add caching for read-heavy operations. The main goal would be to reduce pressure on the database while keeping credit safety intact.
